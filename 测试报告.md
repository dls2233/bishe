# 第 X 章 系统测试与分析

## X.1 测试环境与工具

本系统后端测试采用 `JUnit 5` 与 `Mockito` 框架进行单元测试，采用 Spring Boot Test 的 `MockMvc` 进行集成测试。同时，引入了 `JaCoCo` (Java Code Coverage) 插件对测试覆盖率进行自动化统计，确保核心业务逻辑的稳健性。

- **单元测试框架**: JUnit 5, Mockito
- **集成测试框架**: Spring Boot Test (MockMvc)
- **覆盖率统计工具**: JaCoCo
- **构建工具**: Maven

## X.2 核心业务模块单元测试

为了保证系统中敏感操作（如积分扣减、库存扣减等）在并发场景下的绝对安全，本系统针对 Service 层进行了边界条件的黑盒与白盒测试。

以“积分商城兑换” (`GoodsServiceImpl`) 为例，主要设计了以下测试用例，所有用例均已 **100% 通过**：

1. **正常兑换场景 (`testExchangeSuccess`)**：
   - 模拟用户发起兑换请求，校验库存与积分充足。
   - 验证系统按顺序成功调用 `decreasePoints()`、`decreaseStock()` 和 `insertExchangeRecord()`。
   - 验证最终返回状态码为 `200` 及“兑换成功”提示。
2. **异常场景 - 库存不足 (`testExchangeFail_NoStock`)**：
   - 模拟并发下商品库存为 0 的情况。
   - 验证系统立即抛出异常并返回状态码 `400`。
   - **安全断言**：通过 `verify(..., never())` 确保后续的积分扣减 SQL 绝对未被执行，防止出现“黑洞扣费”问题。
3. **异常场景 - 积分不足 (`testExchangeFail_NotEnoughPoints`)**：
   - 模拟用户余额不足，数据库层返回更新行数为 0。
   - 验证系统回滚操作，返回状态码 `400`，并断言商品库存没有被错误扣减。

## X.3 控制器集成测试

本系统利用 `@AutoConfigureMockMvc` 模拟真实的 HTTP 请求流转，并结合 `@Transactional` 注解保证测试产生的脏数据在测试结束后自动回滚，不污染本地开发数据库。

以“用户登录注册模块” (`UserControllerTest`) 为例，测试执行结果如下（均已通过）：

- **注册流程测试**：构造合法的 JSON 参数请求 `/api/user/register`，期望 HTTP 状态为 200，并验证返回结构中 `code=200`。
- **并发重复注册测试**：模拟恶意用户连续发送两次相同的注册请求，验证系统依靠数据库唯一约束正确拦截，第二次请求返回 `code=400` 及“用户名已存在”。
- **安全登录测试**：使用合法账号请求 `/api/user/login`，断言系统正确下发了符合签名的 JWT Token 字符串。
- **密码校验测试**：输入错误的密码，断言拦截器与控制层协同工作，返回 `code=401` 及“密码错误”。
